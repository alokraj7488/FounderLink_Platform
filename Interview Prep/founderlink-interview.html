Requirement Understanding

FOUNDATION

Problem Statement

India's startup ecosystem is fragmented. Founders struggle to find investors, co-founders, and team members. Investors have no structured way to discover vetted startups. There is no unified platform that handles discovery, team formation, investment, and real-time communication under one roof.

💡
FounderLink — The Solution
A full-stack microservices-based startup ecosystem platform

FounderLink is a platform that connects Founders, Co-Founders, and Investors on a single platform. It allows founders to register startups, investors to discover and invest in startups, team members to collaborate, and admins to govern the ecosystem — all with real-time notifications, secure payments, and event-driven architecture.

Config Server

Purpose: Centralized configuration management. All 11 microservices pull their application.yml properties from this server on startup instead of bundling config inside their JARs. This enables zero-downtime config changes.
JAVAConfigServerApplication.java
@SpringBootApplication          // Enables auto-config, component scan, property loading
@EnableConfigServer             // Activates Spring Cloud Config Server magic
public class ConfigServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(ConfigServerApplication.class, args);
    }
}
🔑 How it works internally

@EnableConfigServer tells Spring to activate a EnvironmentController that exposes REST endpoints like /auth-service/default. When any microservice starts with spring.config.import=configserver:http://localhost:8888, it makes an HTTP GET to this endpoint and receives all the YAML properties. The Config Server in this project reads YAMLs from the local Git-Config-Repo folder.

Eureka Server (Service Registry)

Purpose: Acts as a phone directory for microservices. When StartupService wants to call UserService, it doesn't need a hardcoded IP. It asks Eureka: "Where is user-service?" Eureka returns the current host:port.
JAVAEurekaServerApplication.java
@SpringBootApplication
@EnableEurekaServer     // Turns this app into a Eureka registry dashboard
public class EurekaServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(EurekaServerApplication.class, args);
    }
}
🔑 Eureka Internals

Registration: Each microservice (with @EnableDiscoveryClient or spring-cloud-starter-netflix-eureka-client dependency) sends a heartbeat every 30 seconds to Eureka. If 3 heartbeats are missed, Eureka removes the instance.

Discovery: OpenFeign clients call Eureka to resolve startup-service to an actual IP. Load balancing (via Spring Cloud LoadBalancer) distributes calls across multiple instances.

API Gateway

Purpose: Single entry point for all frontend requests. Routes traffic to appropriate microservices, validates JWT tokens before forwarding requests, adds X-User-Id and X-User-Roles headers, handles CORS.
JAVAJwtAuthenticationFilter.java — API Gateway
@Component
public class JwtAuthenticationFilter 
        extends AbstractGatewayFilterFactory<Config> {

    private static final List<String> PUBLIC_PATHS = 
        List.of("/auth/register", "/auth/login", "/auth/refresh");

    @Override
    public GatewayFilter apply(Config config) {
        return (exchange, chain) -> {
            String path = exchange.getRequest().getURI().getPath();

            // 1. Skip token check for public endpoints
            if (isPublicPath(path)) return chain.filter(exchange);

            // 2. Extract Bearer token from Authorization header
            String authHeader = request.getHeaders().getFirst(AUTHORIZATION);
            if (authHeader == null || !authHeader.startsWith("Bearer ")) {
                exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
                return exchange.getResponse().setComplete();
            }

            // 3. Validate JWT and extract claims
            String token = authHeader.substring(7);
            Claims claims = extractAllClaims(token); // throws on invalid/expired

            // 4. Forward enriched request to downstream service
            ServerHttpRequest modifiedReq = request.mutate()
                .header("X-User-Id", claims.get("userId").toString())
                .header("X-User-Roles", String.join(",", roles))
                .build();

            return chain.filter(exchange.mutate().request(modifiedReq).build());
        };
    }
}
⚡ Critical Interview Point — Why does Gateway validate JWT?

The API Gateway validates the JWT token itself (not by calling Auth Service) because it has the same JWT secret injected via Config Server. This avoids a network round-trip to Auth Service on every request — dramatically improving latency. Once validated, the Gateway adds X-User-Id and X-User-Roles as trusted headers so downstream services know the authenticated user without re-validating the token.

Auth Service

Purpose: Handles user registration, login, JWT issuance, and token refresh. Publishes user.registered event to RabbitMQ after registration.

AuthController.java — Detailed Breakdown

JAVAAuthController.java
@RestController              // = @Controller + @ResponseBody: returns JSON, not views
@RequestMapping("/auth")    // All endpoints in this class are prefixed with /auth
public class AuthController {

    // Constructor Injection (preferred over @Autowired — testable, immutable)
    private final IAuthService authService;      // Interface, not impl (loose coupling)
    private final UserRepository userRepository;
    private final AuthMapper authMapper;

    public AuthController(IAuthService authService, ...) {
        this.authService = authService;
        ...
    }

    @PostMapping("/register")  // HTTP POST /auth/register
    public ResponseEntity<ApiResponse<RegisterResponse>> register(
            @Valid @RequestBody RegisterRequest request) {
        // @Valid → triggers Bean Validation (Jakarta @NotBlank etc. on DTO)
        // @RequestBody → deserializes JSON body into RegisterRequest object
        RegisterResponse response = authService.register(request);
        return ResponseEntity.status(HttpStatus.CREATED)   // HTTP 201
            .body(ApiResponse.success("User registered successfully", response));
    }

    @PostMapping("/login")
    public ResponseEntity<ApiResponse<AuthResponse>> login(
            @Valid @RequestBody LoginRequest request) {
        AuthResponse response = authService.login(request);
        return ResponseEntity.ok(ApiResponse.success("Login successful", response));
    }

    @GetMapping("/users/{id}")  // Used by other services to fetch user info
    public ResponseEntity<UserSummaryDto> getUserById(@PathVariable Long id) {
        return userRepository.findById(id)        // Returns Optional<UserEntity>
            .map(u -> ResponseEntity.ok(authMapper.toUserSummaryDto(u)))
            .orElse(ResponseEntity.notFound().build());
    }
}

AuthService.java — Business Logic

JAVAAuthService.java
@Service    // Marks as Spring-managed business logic bean
@Slf4j      // Lombok: injects private static final Logger log = LoggerFactory.getLogger(AuthService.class)
public class AuthService implements IAuthService {

    @Transactional              // Wraps entire method in a DB transaction. Rolls back on RuntimeException.
    public RegisterResponse register(RegisterRequest request) {

        // Guard 1: Block admin self-registration
        if ("ROLE_ADMIN".equalsIgnoreCase(request.getRole()))
            throw new CustomException("Admin cannot self-register", HttpStatus.FORBIDDEN);

        // Guard 2: Check email uniqueness
        if (userRepository.existsByEmail(request.getEmail()))
            throw new CustomException("Email already registered", HttpStatus.CONFLICT);

        // Fetch role entity or fail with meaningful error
        RoleEntity role = roleRepository.findByName(request.getRole())
            .orElseThrow(() -> new CustomException("Invalid role", HttpStatus.BAD_REQUEST));

        // Build entity using Lombok @Builder pattern
        UserEntity user = UserEntity.builder()
            .name(request.getName())
            .email(request.getEmail())
            .password(passwordEncoder.encode(request.getPassword())) // BCrypt hash
            .roles(Set.of(role))
            .build();

        UserEntity savedUser = userRepository.save(user);  // JPA INSERT + flush

        // Send welcome email synchronously
        welcomeEmailService.sendWelcome(savedUser.getEmail(), ...);

        // Publish async event (fire-and-forget, non-blocking)
        try {
            rabbitTemplate.convertAndSend(exchange, "user.registered",
                UserRegisteredEvent.builder().userId(savedUser.getId())....build());
        } catch (Exception e) {
            log.warn("RabbitMQ publish failed — continuing anyway"); // graceful degradation
        }
    }
}
🧠 Why is the RabbitMQ call in a try-catch?

This is called graceful degradation. The RabbitMQ event is not critical to the registration transaction. If RabbitMQ is down, we don't want to fail the user's registration. We log a warning and move on. The user is still created in the DB. This is a common real-world pattern in event-driven systems.

UserEntity.java — JPA Entity

JAVAUserEntity.java
@Entity              // Marks as JPA-managed class. Hibernate creates/manages the table.
@Table(name = "users") // Maps to the "users" table in PostgreSQL
@Data                // Lombok: generates getters, setters, toString, equals, hashCode
@Builder             // Lombok: enables UserEntity.builder().name("x").email("y").build()
@NoArgsConstructor   // Lombok: required by JPA (JPA needs a no-arg constructor to instantiate)
@AllArgsConstructor  // Lombok: used by @Builder internally
public class UserEntity {

    @Id                                        // Primary key field
    @GeneratedValue(strategy = GenerationType.IDENTITY) // DB auto-increment
    private Long id;

    @Column(nullable = false)               // NOT NULL constraint in DB
    private String name;

    @Column(nullable = false, unique = true)// UNIQUE constraint — prevents duplicate emails
    private String email;

    @ManyToMany(fetch = FetchType.EAGER)     // Load roles immediately with user (no lazy loading)
    @JoinTable(
        name = "user_roles",                 // Creates a join table in DB
        joinColumns = @JoinColumn(name = "user_id"),
        inverseJoinColumns = @JoinColumn(name = "role_id")
    )
    @Builder.Default
    private Set<RoleEntity> roles = new HashSet<>();

    @PrePersist                              // JPA lifecycle hook: called just before INSERT
    protected void onCreate() {
        this.createdAt = LocalDateTime.now();
    }
}

Startup Service

Purpose: Core domain service. Manages startup lifecycle: creation, admin approval/rejection, search with filters, follower management, and publishing events when startups are created/deleted/rejected.

StartupController.java — Key Patterns

JAVAStartupController.java
@RestController
@RequestMapping("/startups")
public class StartupController {

    // CQRS pattern: separate Command and Query services
    private final StartupCommandService startupCommandService;  // Writes (CREATE/UPDATE/DELETE)
    private final StartupQueryService startupQueryService;      // Reads (GET/SEARCH)

    @PostMapping
    @PreAuthorize("hasAuthority('ROLE_FOUNDER')")  // Method-level auth. Only founders can create.
    public ResponseEntity<StartupResponse> createStartup(
            Authentication authentication,         // Injected by Spring Security — current user
            @Valid @RequestBody StartupRequest request) {
        Long founderId = Long.parseLong(authentication.getName()); // getName() returns X-User-Id
        StartupResponse response = startupCommandService.createStartup(founderId, request);
        return new ResponseEntity<>(response, HttpStatus.CREATED);
    }

    @GetMapping
    public ResponseEntity<Page<StartupResponse>> getAllApprovedStartups(
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "10") int size) {
        Pageable pageable = PageRequest.of(page, size);  // Pagination object
        return ResponseEntity.ok(startupQueryService.getAllApprovedStartups(pageable));
    }

    @PutMapping("/{id}/approve")
    @PreAuthorize("hasAuthority('ROLE_ADMIN')")  // Only admins can approve
    public ResponseEntity<StartupResponse> approveStartup(@PathVariable Long id) {
        return ResponseEntity.ok(startupCommandService.approveStartup(id));
    }
}

Startup Entity

JAVAStartup.java
@Entity
@Table(name = "startups")
@Getter @Setter @Builder @NoArgsConstructor @AllArgsConstructor
public class Startup {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(columnDefinition = "TEXT")  // TEXT type in PostgreSQL for long strings
    private String description;

    @Enumerated(EnumType.STRING)  // Stores enum name ("SEED","SERIES_A") not ordinal (0,1)
    @Column(nullable = false)
    private StartupStage stage;

    @Column(nullable = false, precision = 15, scale = 2)  // DECIMAL(15,2) in DB
    private BigDecimal fundingGoal;

    @Builder.Default
    private Boolean isApproved = false; // New startups are pending by default

    @CreationTimestamp  // Hibernate sets automatically on INSERT
    @Column(updatable = false)
    private LocalDateTime createdAt;

    @UpdateTimestamp    // Hibernate updates automatically on every UPDATE
    private LocalDateTime updatedAt;
}

🔑 How OpenFeign works internally

At startup, Spring scans for @FeignClient interfaces. It creates a dynamic proxy (using Java's Proxy API) that implements the interface. When you call startupClient.getStartupById(1L), the proxy intercepts it, looks up startup-service in Eureka, picks an available instance, builds an HTTP GET request to /startups/1, executes it via an HTTP client (default: OkHttp), and deserializes the JSON response into a StartupDTO. You write zero HTTP boilerplate.

Payment Service (Saga Pattern)

Purpose: Handles Razorpay order creation and payment verification. Implements the Saga Orchestration Pattern to manage multi-step distributed transactions with compensation logic.
💡 Saga Pattern — The Core Idea

In microservices, you can't use a single database transaction across multiple services. Instead, a Saga is a sequence of local transactions. If any step fails, compensating transactions are executed in reverse to undo the changes.

In FounderLink's payment flow:

ORDER_CREATED → PAYMENT_CAPTURED → AWAITING_APPROVAL → ACCEPTED ✅

Compensation flow (on rejection):

COMPENSATING → Razorpay Refund → Publish REJECTED event → COMPENSATED ✅

        notificationService.createNotification(event.getInvestorId(), ...);
        notificationService.createNotification(event.getFounderId(), ...);
    }
}

User Service

Purpose: Manages user profiles (separate from auth credentials). Listens to user.registered event and auto-creates a profile. Exposes endpoints for profile CRUD operations.
🔑 Why is User Service separate from Auth Service?

Auth Service only cares about credentials (email, password, role, JWT). User Service manages profile data (bio, location, skills, LinkedIn URL, profile picture). This is the Separation of Concerns principle. If we change the auth mechanism tomorrow (e.g., move to OAuth2), User Service doesn't change at all.

Spring MVC Annotations

AnnotationDefinitionInternal Working
@RestController Combination of @Controller + @ResponseBody DispatcherServlet routes HTTP requests to this class. @ResponseBody tells Spring to serialize the return value to JSON using Jackson's ObjectMapper, instead of looking for a view (Thymeleaf/JSP).
@RequestMapping Maps HTTP requests to handler classes/methods Spring's HandlerMapping registers this path-to-method binding during application startup. URL patterns are stored in a Map for O(1) lookup at runtime.
@GetMapping / @PostMapping / @PutMapping / @DeleteMapping HTTP method-specific shortcuts for @RequestMapping Composed annotations. @GetMapping = @RequestMapping(method = RequestMethod.GET). DispatcherServlet checks the HTTP method AND path to route to the right handler.
@PathVariable Extracts a value from the URI path Spring's DefaultConversionService converts the String segment from the URL to the declared type (String, Long, etc.). Example: /startups/{id} → @PathVariable Long id.
@RequestBody Deserializes HTTP request body into a Java object Spring uses HttpMessageConverter (Jackson by default) to read the Content-Type (application/json) and deserialize the JSON string into the annotated parameter type.
@RequestParam Extracts query parameter from URL Reads from the query string. /startups?page=0&size=10 → @RequestParam int page. Can have defaultValue to avoid null.
@Valid Triggers Bean Validation on the annotated parameter Spring delegates to Hibernate Validator. It checks constraints like @NotBlank, @Email, @Size on the DTO fields. If any fail, MethodArgumentNotValidException is thrown and handled by @ExceptionHandler.
@PreAuthorize Method-level security expression evaluated before execution Spring AOP intercepts the call before the method runs. SecurityContextHolder provides the Authentication object. SpEL expression like hasAuthority('ROLE_FOUNDER') checks authorities. Throws AccessDeniedException if false.

JPA / Hibernate Annotations

AnnotationDefinitionSQL Mapping
@Entity Marks class as a JPA-managed persistent entity Hibernate creates/manages a DB table for this class. Class name → table name (by default).

@CreationTimestamp / @UpdateTimestamp Hibernate-specific: auto-set on create/update CreationTimestamp → set once on INSERT. UpdateTimestamp → updated on every UPDATE.
@Transactional Wraps method in a database transaction Spring creates a proxy. On entry: opens transaction. On success: COMMIT. On RuntimeException: ROLLBACK. Enables atomicity across multiple DB operations.

Spring Security + JWT Deep Dive

MOST ASKED

Authentication vs Authorization

🔐 Authentication — WHO are you?

Verifying identity. In FounderLink: Login with email + password. On success, Auth Service issues a JWT token containing userId, email, and roles. Every subsequent request carries this token as proof of identity.

🛡️ Authorization — WHAT can you do?

Verifying permissions. In FounderLink: Even if you're authenticated, @PreAuthorize("hasAuthority('ROLE_FOUNDER')") blocks investors from creating startups. The JWT contains roles which are checked on every request.

JWT Structure

JWT = Header.Payload.Signature HEADER (Base64 encoded): { "alg": "HS256", // HMAC-SHA256 signing algorithm "typ": "JWT" } PAYLOAD (Base64 encoded — NOT encrypted, just encoded!): { "userId": 42, "email": "alice@example.com", "roles": ["ROLE_FOUNDER"], "sub": "alice@example.com", // subject "iat": 1703001600, // issued at (Unix timestamp) "exp": 1703088000 // expiration (Unix timestamp) } SIGNATURE: HMACSHA256( base64UrlEncode(header) + "." + base64UrlEncode(payload), SECRET_KEY ) → Only the server with the secret can generate/verify this.

JwtUtil.java — Token Generation & Validation

JAVAJwtUtil.java
@Component
public class JwtUtil {

    // @Value injects values from application.yml or Config Server at startup
    public JwtUtil(
            @Value("${jwt.secret}") String secret,
            @Value("${jwt.access-token-expiration}") long accessTokenExpiration,
            @Value("${jwt.refresh-token-expiration}") long refreshTokenExpiration) { ... }

    private SecretKey getSigningKey() {
        // Decode Base64 secret → raw bytes → HMAC-SHA key
        return Keys.hmacShaKeyFor(Decoders.BASE64.decode(secret));
    }

    public String generateAccessToken(Long userId, String email, String role) {
        return buildToken(
            Map.of("userId", userId, "email", email, "roles", List.of(role)),
            email, accessTokenExpiration);
    }

    private String buildToken(Map<String, Object> claims, String subject, long expiration) {
        return Jwts.builder()
            .claims(claims)               // Custom payload fields
            .subject(subject)             // Standard "sub" field
            .issuedAt(new Date())         // Standard "iat" field
            .expiration(new Date(System.currentTimeMillis() + expiration)) // "exp" field
            .signWith(getSigningKey())    // Sign with HMAC-SHA256
            .compact();                   // Serialize to JWT string "header.payload.sig"
    }

    public boolean validateToken(String token) {
        try {
            extractAllClaims(token);  // Throws if signature invalid or token expired
            return true;
        } catch (Exception e) { return false; }
    }

    private Claims extractAllClaims(String token) {
        return Jwts.parser()
            .verifyWith(getSigningKey())  // Must match signing key — verifies signature
            .build()
            .parseSignedClaims(token)   // Parses + validates signature + expiry
            .getPayload();
    }
}

    const originalRequest = error.config as ... & { _retry?: boolean };

    if (error.response?.status === 401 && !originalRequest._retry) {
      if (isRefreshing) {
        // Queue this request until refresh completes
        return new Promise((resolve, reject) => {
          failedQueue.push({ resolve, reject });
        });
      }

      originalRequest._retry = true;
      isRefreshing = true;

      try {
        const response = await api.post('/auth/refresh', {}, { ... });
        const newToken = response.data.token;
        tokenService.setToken(newToken);
        processQueue(null, newToken);  // Retry all queued requests
        return api(originalRequest);       // Retry original request
      } catch (refreshError) {
        tokenService.clearAll();
        window.location.href = '/login';  // Force logout
      }
    }

35 QUESTIONS
Q1What is a microservices architecture and why did you use it in FounderLink?
Microservices architecture decomposes an application into small, independently deployable services, each owning its own data and business logic. I used it in FounderLink because each domain (Auth, Startup, Investment, Payment) has different scaling requirements — the Startup Service needs more resources during peak browsing, while Payment Service needs to scale during funding rounds. With microservices, I can scale each service independently. Additionally, fault isolation ensures that if the Messaging Service goes down, the Investment flow continues unaffected. Each service uses its own database schema, enforcing bounded contexts from Domain-Driven Design.
Q2Explain the role of API Gateway in your project.
The API Gateway is the single entry point for all client requests. It performs JWT validation — it decodes the JWT token using the shared secret, extracts userId and roles claims, and injects them as X-User-Id and X-User-Roles HTTP headers before forwarding the request to downstream services. This prevents each microservice from re-validating tokens. It also handles routing (based on URL patterns), CORS (via CorsConfig), and request logging (via RequestLoggingFilter). The reactive nature of Spring Cloud Gateway (built on Netty, not Servlet) makes it non-blocking and efficient for high throughput.
Q3How does JWT authentication work in FounderLink?
When a user logs in via POST /auth/login, the AuthService validates credentials using BCryptPasswordEncoder.matches(). On success, JwtUtil.generateAccessToken(userId, email, role) builds a JWT signed with HMAC-SHA256 using a secret key from Config Server. The token contains claims: userId, email, roles, and expiry. The client stores this token in localStorage via Redux/tokenService. For every subsequent request, Axios interceptors attach Authorization: Bearer <token>. The API Gateway validates the signature (without calling Auth Service), extracts claims, and passes them as headers. Downstream services read these headers to know the authenticated user's identity and roles.
Q4What is the Saga Pattern and how did you implement it?
The Saga pattern manages distributed transactions across microservices. Since we can't span a single ACID transaction across multiple services' databases, a Saga breaks the transaction into a sequence of local transactions. In FounderLink's payment flow: Step 1: Razorpay order created (ORDER_CREATED). Step 2: Payment captured by Razorpay (PAYMENT_CAPTURED). Step 3: Awaiting founder approval (AWAITING_APPROVAL). Step 4a: Accepted → COMPLETED. Step 4b: Rejected → compensating transaction executes a Razorpay refund (COMPENSATING → COMPENSATED). The SagaOrchestrator coordinates each step, persisting state in payment_sagas table for durability and audit. If the refund itself fails, the saga is marked FAILED with a failure reason for manual intervention.
Q5How does OpenFeign work internally?
OpenFeign creates a JDK dynamic proxy at application startup for every @FeignClient interface. When startupClient.getStartupById(42L) is called, the proxy: (1) reads the @GetMapping("/startups/{id}") annotation to determine method and URL template. (2) Substitutes path variables. (3) Asks Spring Cloud LoadBalancer to resolve startup-service from Eureka to an actual host:port. (4) Makes an HTTP GET request using the OkHttp/Apache client. (5) Deserializes the JSON response into StartupDTO using Jackson. If the service is down, the FallbackFactory kicks in and returns a safe default. This is circuit breaking via Resilience4j.
Q6Explain the CQRS pattern in StartupService.
CQRS stands for Command Query Responsibility Segregation. In StartupService, I separated write operations (create, update, delete, approve, reject) into StartupCommandService and read operations (get, search, list) into StartupQueryService. The benefits are: Single Responsibility — each service has one reason to change. Performance — query service can be optimized independently (e.g., add caching on reads without affecting writes). Testability — smaller classes are easier to unit test. In the StartupController, the constructor injects both services, and each endpoint delegates to the appropriate one based on whether it reads or writes data.
Q7What is Spring Cloud Config Server and why is it important?
Config Server provides externalized configuration management. Instead of each microservice having its own application.yml baked into the Docker image, they all pull config from a central Git-backed Config Server at startup. The benefits: (1) Single source of truth for all configs. (2) No rebuild needed to change a config value — update Git, restart service. (3) Secrets management — the JWT secret, DB passwords, RabbitMQ credentials are in one place. (4) Environment-specific configs — auth-service.yml, auth-service-prod.yml for different environments. Services use spring.config.import=optional:configserver:http://config-server:8888 to pull their config.
Q8Explain @Transactional and when it rolls back.
@Transactional uses Spring AOP to wrap the method in a database transaction. Spring creates a CGLIB proxy around the bean. When the annotated method is called: (1) TransactionInterceptor opens a DB connection from HikariCP pool. (2) BEGIN TRANSACTION executes. (3) Your method body runs. (4) If method completes normally → COMMIT. (5) If a RuntimeException (unchecked) is thrown → ROLLBACK. Important: it does NOT rollback for checked exceptions by default. Use @Transactional(rollbackFor = Exception.class) if needed. In AuthService.register(), the transaction wraps email uniqueness check + user save + role assignment — if the save fails for any reason, all changes are rolled back atomically.
Q9How does RabbitMQ event-driven communication work in your project?
FounderLink uses a Topic Exchange named founderlink.exchange. When AuthService completes registration, it calls rabbitTemplate.convertAndSend(exchange, "user.registered", event). Jackson2JsonMessageConverter serializes the UserRegisteredEvent POJO to JSON. The exchange routes this message to all queues bound with routing key user.registered. Both the NotificationService and the UserService have queues bound to this key. Their @RabbitListener methods receive the message in separate threads. NotificationService creates a welcome notification; UserService creates an empty user profile. This is the fan-out on topic exchange pattern — one event, multiple consumers, fully asynchronous.
Q10What is the difference between @Bean and @Component?
@Component is a class-level annotation. Spring auto-detects it during component scan and registers the class as a bean. The class itself is the bean definition. @Bean is a method-level annotation inside a @Configuration class. You have explicit control over object creation — you call constructors, pass arguments, configure properties before returning. Use @Bean when: (1) The class is from a third-party library (you can't add @Component to it). Example: @Bean PasswordEncoder → new BCryptPasswordEncoder(). (2) You need conditional creation. (3) You need to pass constructor arguments. In FounderLink, RabbitMQConfig uses @Bean to configure TopicExchange and Jackson2JsonMessageConverter because these require constructor arguments and configuration.
Q11How does Eureka Service Discovery work?
On startup, each microservice (with Eureka client dependency) registers itself with Eureka by sending a POST to /eureka/apps/{appId} with its host, port, and health check URL. It then sends heartbeats every 30 seconds. If Eureka misses 3 consecutive heartbeats, it removes the instance from the registry. When OpenFeign needs to call startup-service, it asks Eureka for all healthy instances of startup-service. Spring Cloud LoadBalancer uses round-robin to pick one. Eureka also maintains a local cache in each client so they can still resolve services if Eureka itself briefly goes down — this is called self-preservation mode.
Q12Explain @PreAuthorize and how it works.
@PreAuthorize("hasAuthority('ROLE_FOUNDER')") uses Spring Security AOP. When the controller method is invoked, a MethodSecurityInterceptor proxy intercepts the call BEFORE execution. It retrieves the Authentication object from SecurityContextHolder (which was populated by the JWT filter). It evaluates the SpEL expression hasAuthority('ROLE_FOUNDER') against the authentication's authorities. If true, execution proceeds. If false, it throws AccessDeniedException, which is caught by the GlobalExceptionHandler and returned as HTTP 403. This is declarative authorization — the security logic is separated from business logic.
Q13Why did you use constructor injection instead of @Autowired?
Constructor injection is the Spring-recommended approach for three reasons: (1) Immutability — injected fields can be final, preventing accidental reassignment and making the object thread-safe. (2) Testability — you can instantiate the class in unit tests without a Spring container: new AuthService(mockRepo, mockEncoder, ...). With @Autowired field injection, you need reflection or @SpringBootTest. (3) Fail-fast — if a required dependency is missing, the application fails to start immediately with a clear error: "No qualifying bean of type 'IAuthService'". With @Autowired, you might get a NullPointerException at runtime.
Q14How does BCrypt password hashing work?
BCrypt is an adaptive hashing function. Unlike MD5/SHA1, it's designed to be slow — this makes brute-force attacks computationally expensive. It automatically generates a random salt (22 characters) and embeds it in the output hash string. So two users with the same password have different hashes. The output format is $2a$10$salthere22charshashhererest. 10 is the work factor — the number of rounds doubles every time the factor increases by 1. You can increase the factor over time as hardware gets faster. passwordEncoder.matches(rawPassword, encodedPassword) extracts the salt, re-hashes the raw password with the same salt, and compares — no need to store plain text.
Q15What is STATELESS session management and why use it?
In FounderLink, .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS)) tells Spring Security to never create an HttpSession. Each request is fully self-contained — the JWT token carries all identity information. This is critical for microservices because: (1) Horizontal scaling — if Session Service is stateful, all requests from a user must go to the same server. With stateless JWT, any instance of a service can handle any request. (2) No server memory wasted on sessions. (3) Works across services — JWT is valid for all 11 microservices using the same secret. The trade-off is JWT cannot be "invalidated" before expiry (unless you implement a blacklist in Redis).
Q16Explain Spring Data JPA's repository query derivation.
Spring Data JPA parses repository method names at application startup to derive SQL queries. The method name findByRolesName(String roleName) is parsed as: find (SELECT), By (WHERE clause), Roles (navigate the roles association with a JOIN), Name (filter on the name field of the joined entity). The generated SQL is approximately: SELECT u.* FROM users u JOIN user_roles ur ON u.id=ur.user_id JOIN roles r ON ur.role_id=r.id WHERE r.name = ?. For complex queries, you can use @Query("SELECT ...") with JPQL or native SQL. The JPA Specification API is used in StartupService for dynamic multi-criteria search (industry, stage, fundingGoal range).
Q17How does the token refresh mechanism work in the frontend?
The Axios response interceptor catches any 401 Unauthorized response. It checks if the request was already a retry (_retry flag). If not, it sets isRefreshing = true (preventing multiple concurrent refresh calls) and calls POST /auth/refresh with the current token. While refreshing, any other failed requests are pushed to failedQueue. On success, processQueue(null, newToken) resolves all queued promises, retrying them with the new token. On failure (refresh token also expired), tokenService.clearAll() removes stored tokens and the user is redirected to /login. This is the silent token refresh pattern — users never see a logout unless their refresh token is truly expired.
Q18What is the difference between TopicExchange, DirectExchange, and FanoutExchange?
DirectExchange routes messages to queues where the binding key exactly matches the routing key. Simple 1-to-1 routing. FanoutExchange ignores routing keys completely and delivers messages to ALL bound queues. Broadcasting. TopicExchange (used in FounderLink) routes based on pattern matching using wildcards: * matches exactly one word, # matches zero or more words. Example: startup.* matches startup.created and startup.deleted. I chose TopicExchange because different consumers need to subscribe to different patterns of events. NotificationService subscribes to startup.* for all startup events, while InvestmentService only subscribes to startup.deleted.
Q19How does Docker Compose manage service startup order?
Docker Compose uses depends_on with condition: service_healthy. Each service defines a healthcheck — for PostgreSQL: pg_isready command. For RabbitMQ: rabbitmq-diagnostics ping. The startup order in FounderLink: (1) PostgreSQL, RabbitMQ, Redis start in parallel. (2) Config Server starts and becomes healthy. (3) Eureka Server starts (after Config Server is healthy). (4) All business microservices start (after Eureka + all infrastructure is healthy). Without this ordering, a microservice might start before its Config Server is ready, failing to load properties, or start before Eureka is ready, failing to register.
Q20Explain the difference between @Entity and @Table.
@Entity tells JPA that this class represents a persistent entity — it should be mapped to a database table. Without @Entity, JPA ignores the class entirely. @Table(name="users") is optional and customizes the table name. Without it, JPA uses the class name converted by the naming strategy (e.g., UserEntity → user_entity). In FounderLink, @Table(name="users") explicitly sets the table to users (not user_entity). You can also use @Table to define indexes: @Table(name="users", indexes = @Index(columnList="email")) which creates a database index on the email column for faster lookups.
Q21What is @Builder.Default and why is it needed?
When using Lombok's @Builder, any field with an inline default value (like private Boolean isApproved = false) is NOT honored when using the builder pattern. Lombok generates a builder that defaults all fields to null/0/false unless set. @Builder.Default tells Lombok to use the specified default value in the builder. In Startup.java, @Builder.Default private Boolean isApproved = false ensures that when you call Startup.builder().name("x").build(), isApproved is false (not null). Without it, isApproved would be null in the database, violating the @Column(nullable=false) constraint.
Q22How does pagination work with Spring Data JPA?
The controller accepts @RequestParam int page and @RequestParam int size. It creates a Pageable pageable = PageRequest.of(page, size) object. This is passed to startupRepository.findAll(pageable) which returns Page<Startup>. Hibernate generates: SELECT * FROM startups ORDER BY id LIMIT 10 OFFSET 0 and a separate SELECT COUNT(*) FROM startups. The Page object contains: the list of entities for this page, total elements count, total pages, current page, and whether it's the first/last page. The frontend uses totalPages to render pagination buttons.
Q23What is the purpose of DTOs and why not return entities directly?
DTOs (Data Transfer Objects) decouple the API contract from the database schema. Reasons not to return JPA entities directly: (1) Security — entities may have sensitive fields (password, internal flags) that should never be exposed in the API. (2) Serialization issues — @ManyToMany with LAZY loading causes LazyInitializationException during Jackson serialization. (3) Circular references — bidirectional relationships cause infinite recursion during JSON serialization. (4) API stability — you can change the DB schema without breaking the API contract. The Mapper classes (using authMapper.toUserSummaryDto()) convert entities to DTOs, keeping these concerns separate.
Q24How does Spring's @Value work?
@Value("${jwt.secret}") uses Spring's PropertySourcesPlaceholderConfigurer. At application startup, Spring loads properties from all registered PropertySources (application.yml, Config Server, environment variables). When a bean is being initialized and a field or constructor parameter has @Value("${jwt.secret}"), Spring resolves the ${jwt.secret} placeholder by looking up the key in its property sources. The resolved String value is injected. If the key doesn't exist, Spring throws IllegalArgumentException unless a default is provided: @Value("${jwt.secret:myDefaultSecret}"). In FounderLink, the JWT secret comes from Config Server's auth-service.yml.
Q25Explain Spring Cloud Gateway's reactive nature.
Spring Cloud Gateway is built on Project Reactor and Netty (non-blocking I/O). Unlike traditional Servlet-based Spring MVC which uses one thread per request, Gateway uses an event loop model. A small number of threads (usually = number of CPU cores) handle thousands of concurrent connections using non-blocking I/O. The GatewayFilter returns a Mono<Void> (reactive stream). chain.filter(exchange) returns a Mono representing the downstream service call. The return exchange.getResponse().setComplete() for rejected requests is also reactive. This makes the Gateway extremely efficient under high concurrency — it doesn't block threads while waiting for downstream responses.
Q26What happens when multiple API calls fail with 401 simultaneously?
This is handled by the token refresh mutex pattern in axiosConfig.ts. When the first 401 response arrives, isRefreshing is set to true and the refresh call is initiated. When the second and third 401 responses arrive before the refresh completes, they see isRefreshing = true and push their resolve/reject callbacks to failedQueue. When the refresh succeeds, processQueue(null, newToken) iterates through failedQueue and resolves each promise with the new token. Each queued request then retries with Authorization: Bearer <newToken>. This ensures only ONE refresh call is made regardless of how many concurrent requests fail simultaneously.
Q27Explain FetchType.EAGER vs LAZY in JPA.
FetchType.EAGER loads the association immediately when the parent entity is loaded, via a JOIN query. In UserEntity, roles are EAGER — when we call userRepository.findByEmail(), Hibernate immediately joins and fetches roles. This is correct here because we always need roles for authentication. FetchType.LAZY (default for @ManyToOne, @OneToMany) creates a proxy that only loads data when you actually access the field. This is better for performance when you don't always need the association. The danger: if you access a LAZY field outside a transaction (e.g., during JSON serialization), Hibernate throws LazyInitializationException because the session is closed. DTOs solve this by fetching needed data within the transaction.
Q28How does the GitHub Actions CI/CD pipeline work?
The pipeline triggers on push or pull_request to main. Job 2 (build-frontend): Sets up Node.js 18, runs npm ci (clean install), linting, TypeScript check, and production build — in parallel with Job 1. Q29What is the CSRF protection and why is it disabled?
CSRF (Cross-Site Request Forgery) is an attack where a malicious site tricks a browser into making authenticated requests using stored cookies. Spring Security adds a CSRF token to HTML forms to prevent this. However, in FounderLink, authentication uses Bearer JWT tokens in the Authorization header, not cookies. CSRF attacks only work when credentials are automatically sent by the browser (cookies). JavaScript code must explicitly set the Authorization header — a cross-site script cannot access another site's localStorage. Therefore, CSRF protection is unnecessary and is disabled with csrf(AbstractHttpConfigurer::disable) to simplify the stateless REST API.
Q30What is @EnumType.STRING and why use it over ORDINAL?
@Enumerated(EnumType.STRING) stores the enum's name (e.g., "SEED") in the database column. EnumType.ORDINAL (the default) stores the position index (0, 1, 2). The problem with ORDINAL: if you insert a new enum value in the middle of the enum declaration, all existing data becomes incorrect. Example: enum StartupStage { IDEA, SEED, SERIES_A } — SEED is stored as 1. If you later add PRE_SEED before SEED: enum StartupStage { IDEA, PRE_SEED, SEED, SERIES_A } — now SEED is stored as 2, but the DB still has 1 for old records. This causes data corruption. STRING is always safe because it stores the actual name regardless of ordering.
Q31Explain the GlobalExceptionHandler pattern.
Each microservice has a @RestControllerAdvice class (GlobalExceptionHandler) that provides centralized exception handling. Instead of try-catch blocks scattered across controllers, exceptions bubble up and are caught here. @ExceptionHandler(ResourceNotFoundException.class) handles 404s. @ExceptionHandler(MethodArgumentNotValidException.class) handles validation failures (from @Valid). @ExceptionHandler(CustomException.class) handles business exceptions with a specific HttpStatus. Each handler returns a structured ApiResponse JSON body with a meaningful error message. This gives consistent error format across all endpoints and separates error-handling logic from business logic.
Q32What is HikariCP and why is it important?
HikariCP is Spring Boot's default JDBC connection pool (since Spring Boot 2.0). Creating a new database connection is expensive (100-200ms). HikariCP maintains a pool of pre-opened connections and reuses them. When @Transactional starts a transaction, it borrows a connection from the pool. On commit/rollback, the connection is returned. HikariCP is the fastest connection pool available for JVM — it uses bytecode-level optimization. Default pool size is 10. If all 10 connections are in use and a new transaction needs one, the caller waits up to connectionTimeout (30s default). Monitoring the hikaricp_connections_active metric in Grafana helps detect connection pool exhaustion.
Q33How does the ProtectedRoute work in React?
ProtectedRoute is a wrapper component that checks authentication before rendering children. It reads isAuthenticated and role from the useAuth hook (which reads Redux state). If isAuthenticated is false or the token is missing from localStorage, it clears all tokens and navigates to /login using React Router's <Navigate>. If allowedRoles is provided and the user's role isn't in the list, it navigates to /unauthorized. This implements both authentication guard (is user logged in?) and authorization guard (does user have the right role?). The replace prop means the login page replaces the current history entry so pressing Back doesn't go back to the protected page.
Q34Explain multi-stage Docker builds and their benefit.
Multi-stage builds use multiple FROM statements in a Dockerfile. Stage 1 (build): Uses a full JDK image to compile the code and build the JAR. Stage 2 (runtime): Copies only the built JAR from Stage 1 into a minimal JRE image. The final Docker image contains only what's needed to run — no Maven, no source code, no JDK compiler. Benefits: (1) Smaller image size — JRE image is ~100MB vs JDK image ~350MB. Smaller images mean faster deployment, less storage, faster Docker pulls. (2) Security — no compilation tools in production means a smaller attack surface. (3) Build reproducibility — build always happens inside Docker, not on developer's machine.
Q35What is Spring Boot Actuator and what endpoints does it expose?
Spring Boot Actuator adds production-ready monitoring endpoints to every microservice. Key endpoints: /actuator/health — reports UP/DOWN status (used by Docker healthcheck). /actuator/prometheus — Prometheus-format metrics scraped every 15s by Prometheus server. /actuator/info — application metadata. /actuator/metrics — lists all available metrics. /actuator/env — current environment properties (disabled in prod for security). In FounderLink, /actuator/** is permitAll() in SecurityConfig so Docker healthchecks and Prometheus scraping work without authentication. Actuator integrates with Micrometer for metric collection — counters, gauges, timers are automatically registered for JVM, Hikari, HTTP requests, etc.

S1What happens if RabbitMQ goes down while a user is registering?
The rabbitTemplate.convertAndSend() call in AuthService.register() is wrapped in a try-catch. If RabbitMQ is unavailable, a AmqpConnectException is thrown. The catch block logs a warning but does NOT re-throw the exception. The registration transaction continues and commits — the user IS saved in the database. They just won't get a welcome notification or have their UserService profile created immediately. This is graceful degradation. A production improvement would be to use a transactional outbox pattern — save the event to a DB table inside the same transaction, then a background job polls and publishes events, guaranteeing at-least-once delivery.
S2How would you handle if the Startup Service is down when Team Service tries to validate a startup?
The @FeignClient on StartupClient has a fallbackFactory. When Startup Service is unavailable, the fallback returns null. The Team Service's sendInvitation method checks if the returned startup DTO is null and throws a ServiceUnavailableException, which is mapped to HTTP 503 by the GlobalExceptionHandler. For improvement, I'd add Resilience4j Circuit Breaker: after 5 consecutive failures, the circuit "opens" and immediately returns the fallback for 30 seconds without making network calls, preventing cascading failures and giving Startup Service time to recover.
S3If the platform needs to scale to 10,000 concurrent users, what would you change?
Multiple improvements: (1) Horizontal scaling — run multiple instances of high-traffic services (StartupService, AuthService) behind a load balancer. Eureka + Ribbon handles this automatically. (2) Redis caching for frequently read data (approved startup listings) with TTL to reduce DB load. (3) Database read replicas — route SELECT queries to read replicas, writes to primary. (4) Increase HikariCP pool size per instance. (5) Async processing — already using RabbitMQ for notifications, extending to more operations. (6) CDN for static React assets. (7) Kubernetes with HPA (Horizontal Pod Autoscaler) for auto-scaling based on CPU/memory metrics visible in Grafana.
S4How would you handle a failed Razorpay refund in the Saga?
In the SagaOrchestrator.compensate() method, if the Razorpay refund call fails (network error, Razorpay API issue), the exception is caught and logged. The saga is marked SagaStatus.FAILED with a failureReason ("Razorpay refund failed: <error message>"). The event rejection notification is STILL published (non-blocking) so both parties are informed. The payment_sagas table entry with status FAILED and the failure reason enables a support team to manually process the refund. A production improvement: implement a retry mechanism with exponential backoff for the refund call, and an admin dashboard to view and manually trigger failed saga compensations.
S5How would you prevent a founder from seeing other founders' pending startups?
The GET /startups endpoint only returns approved startups (isApproved=true). The GET /startups/admin/all endpoint is @PreAuthorize("hasAuthority('ROLE_ADMIN')"). The GET /startups/founder/{founderId} endpoint returns all startups for a specific founderId. In the implementation, the service checks if authentication.getName() (the requesting user's ID) matches founderId in the path — a founder can only see their own pending startups. If they request another founder's ID, they get a 403. Additionally, the filter chains in each service validate the JWT headers injected by the Gateway, preventing unauthorized header manipulation.
S6What would you do if two founders accidentally submit the same startup name?
Currently, there's no unique constraint on startup name in the schema (startups from different founders could have the same name). To prevent duplicate submissions from the same founder, I'd add a composite unique constraint: @Table(uniqueConstraints = @UniqueConstraint(columnNames = {"name", "founderId"})). If violated, Hibernate throws DataIntegrityViolationException, which the GlobalExceptionHandler catches and converts to a 409 Conflict response. For platform-wide uniqueness, a simple unique constraint on name with a DuplicateResourceException in the service layer checked before saving would work.
S7How would you add an audit trail for all admin actions?
I'd use Spring AOP with a @Around advice (similar to the existing LoggingAspect in each service). Create an @Audit annotation. Apply it to admin-only methods. The aspect intercepts these calls, extracts the Authentication from SecurityContextHolder, logs the admin's userId, the action performed, the entity affected, timestamp, and the before/after state to a dedicated audit_log table. This is the Aspect-Oriented Programming approach — audit logic is centralized and doesn't pollute business methods. Spring AOP is used throughout the project in LoggingAspect.java in every service.
S8How do you ensure the JWT secret is not exposed?
The JWT secret is stored in the Config Server's Git repository. In production, this Git repo would be private. For extra security: (1) The Config Server itself can encrypt sensitive properties using spring.cloud.config.server.encrypt.key — values are stored as {cipher}encryptedValue in YAML. (2) Environment variables can override config values — Docker Compose passes secrets from a .env file (which is in .gitignore). (3) For production Kubernetes, use Kubernetes Secrets or HashiCorp Vault as a Config Server backend. The .env file in the project root is already in .gitignore to prevent accidental commit.
S9If the notification service is slow, how would it affect other services?
Because RabbitMQ provides temporal decoupling, a slow NotificationService does NOT affect AuthService, StartupService, or any other service. When AuthService publishes a user.registered event, it immediately continues. The message sits in the RabbitMQ queue. NotificationService consumes it when it's ready, even if that's 10 seconds later. The queue acts as a buffer. This is the core advantage of async event-driven architecture over synchronous REST calls. If NotificationService uses synchronous calls instead, a slowdown there would block the entire registration flow. The queue depth metric in Prometheus/Grafana would alert us if NotificationService falls behind.
S10How would you implement rate limiting on the API Gateway?
Spring Cloud Gateway supports rate limiting with Redis. Add the RequestRateLimiter filter in GatewayConfig: configure a RedisRateLimiter with replenishRate (tokens per second) and burstCapacity (max burst). Use a KeyResolver (e.g., by IP address or X-User-Id header) to identify rate-limited subjects. Redis stores token bucket state. When a request arrives, the filter checks Redis. If tokens available: allow request, decrement token. If exhausted: return HTTP 429 Too Many Requests. This is the Token Bucket algorithm. Redis is already in the architecture so no new infrastructure is needed.
S11How would you debug a 401 error that only happens intermittently?
Steps: (1) Check the API Gateway logs — does the JWT filter reject the request? (2) Verify the JWT expiry — the token may be expiring during the session (access token might have a short expiry in Config Server). (3) Check Grafana for request latency — if the Gateway takes too long, the access token might expire in transit. (4) Check if it correlates with specific services — if only Startup Service returns 401, perhaps its local JWT filter has a different secret than the Gateway. (5) Use Zipkin distributed tracing — trace the request ID across services to see exactly where authentication fails. (6) Enable logging.level.org.springframework.security=DEBUG temporarily to see Spring Security's decision-making.
S12How would you migrate the database schema without downtime?
Use Flyway or Liquibase for database migration management. Add migration scripts (V1__create_users.sql, V2__add_startup_location.sql). On service startup, Flyway runs pending migrations in order. For zero-downtime: (1) Additive changes first — add new column as nullable (old code ignores it, new code uses it). (2) Deploy new code. (3) Non-null/constraints — only after all instances run new code. This is the expand-contract pattern. For the current project, spring.jpa.hibernate.ddl-auto=validate in production (not create-drop) ensures Hibernate validates the schema without modifying it, while Flyway handles migrations.
S13How would you handle a distributed transaction spanning Payment and Investment services?
This is exactly what the Saga pattern solves. When a payment is accepted, InvestmentService needs to update the investment status. Options: (1) Choreography-based Saga (current approach for notifications): PaymentService publishes payment.accepted event. InvestmentService subscribes and updates its state. No central coordinator. (2) Orchestration-based Saga (the SagaOrchestrator in PaymentService): Explicitly calls each step in sequence and handles rollbacks. Better visibility. I chose orchestration for payments because the payment flow has strict ordering (can't mark investment successful before payment is captured) and compensation logic (refund) that's difficult to implement with pure choreography.
S14A new requirement: investors can message founders directly. How do you implement it?
The MessagingService already exists (port 8086) with WebSocket support. Architecture: (1) MessagingService stores messages in messaging_db table (sender_id, receiver_id, content, timestamp, startup_id for context). (2) REST endpoint POST /messages to send a message and GET /messages/conversation/{userId} to fetch history. (3) WebSocket endpoint for real-time delivery — when a message arrives, if recipient is connected, push via WebSocket; otherwise store for later retrieval. (4) Frontend useNotificationSocket hook already handles WebSocket connections. (5) @PreAuthorize ensures only authenticated users can send. (6) Publish message.sent event to NotificationService for push notification when user is offline.
S15How would you implement startup search with multiple dynamic filters?
StartupService already uses JPA Specifications (StartupSpecification) for dynamic querying. Specification<Startup> is a functional interface that builds a JPA Predicate. For each non-null filter parameter (industry, stage, minFunding, location), a separate Specification is built and combined with Specification.where(s1).and(s2).and(s3). Hibernate generates a single SQL query with a WHERE clause containing only the provided conditions. Example: searching by industry=fintech&stage=SEED&minFunding=100000 generates WHERE industry='fintech' AND stage='SEED' AND funding_goal >= 100000 AND is_approved=true. This is more maintainable than building query strings manually.
S16Describe how you'd handle secrets in a production environment.
For production, I'd use HashiCorp Vault as a Config Server backend. Vault stores secrets encrypted at rest, supports fine-grained access policies, and has audit logging for every secret access. Spring Cloud Config Server has native Vault support. Alternatively, on AWS: store secrets in AWS Secrets Manager and reference them in ECS task definitions as environment variables. Kubernetes: use Kubernetes Secrets with RBAC policies so only specific service accounts can read the JWT secret. CI/CD: GitHub Secrets for Docker Hub credentials (already in use in the ci-cd.yml). Never commit secrets to Git — the .env file is gitignored, and the Config Server's Git repo with real secrets should be private.
S17What happens if two instances of TeamService try to send the same invitation?
There's a unique constraint on (startup_id, invited_user_id) in the team_invitations table (as a DuplicateResourceException guard in the service layer). If two instances race to create the same invitation, both hit the service layer check. If both pass the check (race condition window), Hibernate will try to INSERT both. The database constraint will reject the second INSERT, throwing DataIntegrityViolationException. The service catches this and throws DuplicateResourceException → HTTP 409. For stronger consistency: use @Transactional with SELECT ... FOR UPDATE (pessimistic locking) or database-level unique constraints as the final guard. This is the optimistic vs pessimistic locking consideration in distributed systems.
S18How would you monitor and alert on payment failures?
Several layers: (1) Prometheus metrics — create a custom counter payment.saga.failures_total using Micrometer, incremented whenever a saga enters FAILED state. Grafana alert rule fires when count increases. (2) RabbitMQ queue depth — if payment-failed-queue depth suddenly spikes, something is wrong. (3) payment_sagas table — SagaStatus.FAILED rows need immediate attention. An admin dashboard query showing FAILED sagas with failure reasons. (4) Email alerts — the EmailService in PaymentService is already set up; add an alert email to the ops team when saga fails. (5) Zipkin tracing — trace payment flows end-to-end to identify where failures occur.
S19How would you implement search functionality across all startups efficiently?
Current implementation uses JPA Specifications with SQL LIKE queries — acceptable for small datasets. For scale: (1) Elasticsearch — index startup documents (id, name, description, industry, stage, fundingGoal, location) in Elasticsearch. StartupCommandService publishes an event after each create/update. A new Search Service subscribes and indexes the document. Elasticsearch provides full-text search, aggregations, and geographic queries. (2) Alternatively, PostgreSQL Full-Text Search with tsvector columns and GIN indexes — significantly faster than LIKE queries, no extra infrastructure. (3) Short-term: add database indexes on industry and stage columns if not already present.
S20How would you version your APIs?
API versioning options: (1) URL versioning (simplest): /api/v1/startups, /api/v2/startups. API Gateway routes based on version prefix. (2) Header versioning: Accept: application/vnd.founderlink.v2+json. @RequestMapping(headers="X-API-Version=2") in controllers. (3) Content negotiation: Different DTOs for v1 and v2. For FounderLink, I'd use URL versioning — it's visible, bookmarkable, and easy to route at the Gateway. Migration path: run v1 and v2 simultaneously for 3 months, deprecate v1, then remove. The API Gateway's routing config simply maps /v1/** to old service endpoints and /v2/** to new ones, enabling gradual migration without downtime.

Power Interview Sentences

IMPRESS THE INTERVIEWER

Memorize these and use them confidently. They signal deep understanding, not just surface knowledge.

I designed the API Gateway to validate JWT tokens locally using the shared HMAC secret, rather than calling the Auth Service on every request — this eliminated a synchronous network hop from the critical path and reduced authentication latency to near-zero.
I applied the CQRS pattern in StartupService by separating CommandService for writes and QueryService for reads. This isn't premature optimization — it enforces Single Responsibility, enables independent caching strategies for reads, and makes the codebase significantly easier to maintain as the startup domain grows.
The RabbitMQ event publishing in AuthService's register() method is intentionally wrapped in a try-catch because we practice graceful degradation — the user registration is more critical than the welcome notification, so a transient RabbitMQ failure shouldn't roll back a successful registration.
I chose TopicExchange over FanoutExchange because I needed selective consumption — NotificationService subscribes to all events using wildcard patterns, while InvestmentService only subscribes to startup.deleted to clean up related data. FanoutExchange would force all consumers to handle all messages.
The Saga orchestration in PaymentService persists each step transition in the payment_sagas table. This gives us durability — if the service restarts mid-saga, we can resume from the last known step rather than starting over. The FAILED state with a failure_reason field is essentially an audit trail for manual intervention when compensating transactions fail.
Constructor injection with final fields is not just a stylistic choice — it makes the object genuinely immutable after construction, eliminates field injection's hidden null risks, and allows me to write meaningful unit tests without a Spring context by simply calling new AuthService(mockRepo, mockEncoder, ...).
The token refresh logic in the Axios interceptor uses a mutex pattern with a failedQueue to handle multiple concurrent 401 errors. This guarantees that if 10 API calls fail simultaneously, exactly ONE refresh request is made. The other 9 queue up and retry with the fresh token — users experience this as completely seamless.
@Enumerated(EnumType.STRING) is a deliberate choice over ORDINAL. With ORDINAL, inserting a new enum value anywhere in the declaration corrupts all existing database records. STRING stores the actual name, making schema evolution safe regardless of enum ordering changes.
The Docker Compose healthcheck pattern with depends_on: condition: service_healthy isn't just convenience — it guarantees deterministic startup order across 15+ containers. Config Server must be healthy before Eureka starts, Eureka must be healthy before business services start. Without this, services fail silently due to missing configurations.
Disabling CSRF is correct and intentional for a stateless JWT API. CSRF attacks exploit the browser's automatic cookie-sending behavior. Since our authentication lives in the Authorization header, which JavaScript must set explicitly, cross-site requests cannot carry our credentials. Spring Security's CSRF protection would only add unnecessary complexity without providing any security benefit.
The fallbackFactory pattern in OpenFeign gives us control over the failure response — we get access to the actual Throwable that caused the failure, letting us log differently for a timeout versus a connection refused versus an HTTP 500. A static fallback would hide this context.
SessionCreationPolicy.STATELESS combined with JWT means any of the N instances of StartupService can handle any request from any user. There's no session affinity requirement. This is the foundational principle that makes horizontal scaling in microservices straightforward.
The GitHub Actions matrix strategy builds 11 Docker images in parallel — reducing CI pipeline time from what would be a sequential 20+ minutes to approximately 3-4 minutes. Each image is tagged with both :latest and the git commit SHA, giving us immutable, traceable versioning alongside convenient deployment via :latest.
I use @Builder.Default on boolean flags like isApproved and isRejected because Lombok's @Builder ignores inline initializers by default. Without @Builder.Default, a new Startup built via the builder pattern would have isApproved=null, which would immediately violate the @Column(nullable=false) constraint on save — a silent, hard-to-debug failure.
The PostgreSQL init.sql file in the docker-compose setup creates all separate schemas — auth_db, startup_db, team_db etc. — within a single PostgreSQL instance for development convenience. In production, each microservice would have its own isolated PostgreSQL instance, enforcing true database-per-service isolation — a core principle of microservices architecture.
